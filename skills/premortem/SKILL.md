# Premortem

プレモーテムはポストモーテムの逆です。何かが失敗したあとに何が悪かったのかを調べるのではなく、始める前にすでに失敗したと想像し、その理由を明らかにします。

この手法は心理学者 Gary Klein に由来します。彼はこの手法を Harvard Business Review で発表しました。Daniel Kahneman（「Thinking, Fast and Slow」で知られるノーベル賞受賞心理学者）は、これを自分にとって最も価値のある意思決定技法だと呼びました。Google、Goldman Sachs、Procter & Gamble も大きな意思決定の前にこれを使っています。

核となる洞察はこうです。「何がうまくいかない可能性がありますか？」と聞くと、人は慎重で控えめな答えを出します。一方で「これはすでに失敗しました。理由を教えてください」と言うと、脳は物語モードに切り替わり、はるかに具体的で創造的で正直な理由を生み出します。Wharton と Cornell の研究者はこれを「prospective hindsight」と呼び、将来の結果の原因を特定する能力を大きく高めることを発見しました。

これが AI 支援の意思決定で重要な理由は、多くのAIエージェントが協調的で楽観的な応答をデフォルトにしがちだからです。「これは良い計画ですか？」と聞くと、AIエージェントは「はい」と言う理由を探しやすくなります。プレモーテムは「これは死んだ。どう死んだかを説明せよ」というフレームを強制することで、このパターンを破ります。AIエージェントは計画がうまくいく理由を探すのをやめ、どのように崩壊したのかを説明し始めます。

## いつプレモーテムを実行するか

プレモーテムに適した対象:
- これから作ろうとしているプロダクトや機能
- お金や評判がかかっているローンチ計画
- 価格変更やビジネスモデルの転換
- これから行う採用
- 戦略やポジショニングの転換
- 評価中のパートナーシップや取引
- 間違えた場合のコストが高いコミットメント全般

プレモーテムに適さない対象:
- まだ具体的な計画になっていない曖昧なアイデア（先に計画作りを手伝い、そのあとでプレモーテムする）
- 正解が1つある質問（普通に答える）
- 下書きへの創造的フィードバック依頼（それは編集であり、プレモーテムではない）
- すでに決定済みで取り消せない意思決定（プレモーテムは、まだ進路を変えられるときにだけ有用）

## コンテキスト収集（最低基準）

プレモーテムの質は、それを実行するためのコンテキストの質で決まります。曖昧な入力からは、誰の役にも立たない曖昧な失敗シナリオしか出ません。プレモーテムを実行する前に、最低限のコンテキスト基準を満たす必要があります。

### ステップ 1: 既存のコンテキストを探す

ユーザーに何かを聞く前に、すでに利用できるコンテキストを探してください。

**A. 現在の会話。** ユーザーは、このセッションの前半で計画、ローンチ、プロダクト、意思決定について話しているかもしれません。会話を読み返し、関連するものを抽出します。

**B. ワークスペース。** 関連コンテキストを含んでいそうなファイルを素早く探します。
- `AGENTS.md` などのエージェント向け指示ファイル（ビジネスコンテキスト、好み、制約）
- `memory/` フォルダ（オーディエンスのプロフィール、ビジネス詳細、過去の意思決定）
- ユーザーが明示的に参照または添付したファイル
- プレモーテム対象に関係するプロジェクトファイル、ブリーフ、計画

利用可能なファイル検索と読み取り手段を使います。これに30秒以上かけないでください。失敗シナリオを現実に根ざしたものにするための主要ファイルを探しています。

### ステップ 2: コンテキストが十分か評価する

探索後、有用なプレモーテムを実行するのに十分か確認します。必要なのは3つです。

1. **それは何か？** - プレモーテム対象（プロダクト、ローンチ、採用、価格変更、戦略）を明確に理解していること。ユーザーに対して一文で説明し返せる必要があります。

2. **誰のためか / 誰に影響するか？** - オーディエンス、顧客、チーム、ステークホルダー。失敗シナリオは、関わる人に大きく依存します。

3. **成功とは何か？** - ユーザーが望んでいる成果。失敗は成功を反転させて定義されます。成功の意味がわからなければ、失敗の意味も定義できません。

### ステップ 3: 会話で不足を埋める

3つすべてが揃っているなら、すぐにプレモーテムへ進みます。不要な質問はしないでください。

1つ以上欠けている場合は、最も重要な不足要素をまず質問します。一度に1つだけ質問します。回答を受け取るたびに、もう十分かを評価します。基準を満たすまで質問を続けますが、必要以上に聞いてはいけません。

焦点を絞ったコンテキスト質問の例:
- 「具体的に何をローンチ / 構築 / 決定しようとしていますか？」（それが何か分からない場合）
- 「これは誰向けですか？」（計画は分かっているが、オーディエンスが分からない場合）
- 「これが成功したと言える状態は何ですか？」（計画とオーディエンスは分かっているが、成功条件が分からない場合）

目標は、ユーザーにフォームを埋めさせているように感じさせず、できるだけ早く最低基準に到達することです。尋問ではなく会話的に進めます。コンテキストから推測できるなら、質問せず推測してください。

## プレモーテムセッションの進め方

### ステップ 1: フレームを設定する

十分なコンテキストを集めたら、プレモーテムのフレームを明示的に設定します。たとえば次のようにします。

「OK、十分なコンテキストがあります。プレモーテムを実行します。前提はこうです。今は6か月後です。[計画 / ローンチ / 意思決定] は失敗しました。完全に終わっています。私たちは振り返って、何が悪かったのかを理解しようとしています。」

このフレーミングが重要です。「この計画を評価する」（協調的な応答を誘発する）モードから、「なぜこれが死んだのかを説明する」（正直で具体的な失敗特定を誘発する）モードに切り替えます。

### ステップ 2: 失敗理由を生成する（生のプレモーテム）

生のプレモーテムを、単一の包括的な分析として実行します。規定カテゴリ、レンズ、制約はありません。Klein の中核手法だけを使います。

「この計画は6か月後に失敗しました。これが本当に死に至った可能性のある理由をすべて生成してください。網羅的に、具体的にしてください。すべての理由を計画の実際の詳細に根ざしたものにしてください。弱い理由で水増しせず、まだ理由があるなら早く止まらないでください。」

出力は、失敗理由の包括的なリストにします。各理由は1〜2文で述べます。正直かつ徹底的に行います。計画によっては本物の失敗モードが4つかもしれません。別の計画では9つかもしれません。その数は、この具体的な計画に対して実際に存在するものにします。

各失敗理由は次を満たす必要があります。
- この計画に固有である（何にでも当てはまる一般論ではない）
- ユーザーが提供した実際の詳細に根ざしている
- 本物の脅威である（小さな不便や、極端に起こりにくいエッジケースではない）

### ステップ 3: 深掘りワーカー（失敗理由ごとに1つ、可能なら並列）

ステップ2の各失敗理由について、理由ごとに1つの独立した深掘りワーカーを用意し、並列実行できる環境ではすべて並列で実行します。利用中の環境がサブエージェントを提供している場合は、失敗理由ごとに1つのサブエージェントを起動します。サブエージェントが使えない場合は、失敗理由ごとに独立した分析ブロックを作り、互いの結論を混ぜずに深掘りします。

**深掘りワーカープロンプトテンプレート:**

```
あなたはプレモーテム分析の調査員です。あなたには、深く分析するための具体的な失敗理由が1つ割り当てられています。

計画:
---
[完全なコンテキスト: それが何か、誰向けか、成功とは何か、関連するワークスペースコンテキスト]
---

プレモーテムフレーム: 今は6か月後です。この計画は失敗しました。

あなたに割り当てられた失敗理由: [ステップ2の具体的な失敗理由]

あなたの仕事は、この1つの失敗を深掘りすることです。この具体的な失敗が実際にどう起きたのか、その物語を書いてください。具体的にしてください。計画の詳細を使ってください。実際に起きたことのケーススタディのように、現実味を持たせてください。

出力には次を含めてください。

1. 失敗の物語: この具体的な失敗がどう展開したのかを2〜3段落で語る。計画の詳細を使う。物事が悪化した具体的な瞬間と、その理由を名指しする。

2. 根底にある仮定: この失敗を可能にした、ユーザーが当然視していた1つのこと。一文で述べる。

3. 早期警戒サイン: この失敗モードが起き始めていることを示す、具体的で観察可能なシグナルを1〜2個。曖昧な感覚ではなく、実際に見たり測ったりできるものにする。

全体を短く、300語相当以内を目安にしてください。直接的に書いてください。曖昧にしないでください。砂糖衣をかけないでください。
```

### ステップ 4: 統合

すべての深掘りワーカーが完了したら、すべての深掘りを読み、統合を作成します。

**プレモーテムレポート**

1. **最も起こりやすい失敗** - 計画について分かっていることを踏まえると、どの失敗シナリオが最も起こりやすいか。なぜか。これはユーザーが最初に集中すべきものです。

2. **最も危険な失敗** - 起きる可能性は低くても、起きた場合に最も大きな損害を生む失敗シナリオはどれか。これは保険をかける価値があるものです。

3. **隠れた仮定** - すべての失敗分析を横断して、ユーザーが疑っていなさそうな最大の仮定は何か。プレモーテムの本当の価値は、たいていここにあります。ユーザーにとってあまりに当然なので、それが仮定だと忘れていたものです。

4. **修正版の計画** - 失敗シナリオを踏まえて、どの具体的な変更が計画をより強くするか。具体的にしてください。「価格を検討する」と言ってはいけません。「公にコミットする前に20人に対して $X で価格をテストする」と言ってください。各修正は、具体的な失敗シナリオに直接対応している必要があります。

5. **ローンチ前チェックリスト** - 実行前に検証、テスト、整備すべき具体的な項目を3〜5個。各項目は、特定された失敗モードの1つを防ぐか検出する必要があります。

### ステップ 5: プレモーテムレポートを生成する

視覚的な HTML レポートを生成し、ユーザーのワークスペースに保存します。

**ファイル:** `premortem-report-[timestamp].html`

レポートは、インライン CSS を含む単一の自己完結 HTML ファイルにします。デザイン原則:
- 暗い背景（#0a0e1a または類似色）、読みやすいタイポグラフィ、スキャンしやすい構成
- 統合セクション（最も起こりやすい失敗、最も危険な失敗、隠れた仮定、修正版の計画、チェックリスト）は、ほとんどの人が最初に読む部分なので上部で目立たせる
- 失敗理由ごとに1つのビジュアルカードを用意し、失敗理由をヘッダーとして表示し、失敗の物語、根底にある仮定、早期警戒サインを載せる。各カードには異なるアクセントカラーを使い、視覚的にスキャンしやすくする
- 各失敗モードについて、深刻度 / 可能性の明確な視覚表示を入れる
- ラウンドロビンのビジュアル: 実行した深掘りワーカー数とその発見をグリッドまたはカードレイアウトで示し、ユーザーがプレモーテム全体の範囲を一目で把握できるようにする
- タイムスタンプと、何をプレモーテムしたかを含むフッター

生成後、環境上可能であれば HTML ファイルを開きます。開けない場合は、保存先をユーザーに伝えます。

### ステップ 6: トランスクリプトを保存する

完全なプレモーテムトランスクリプトを、同じ場所に `premortem-transcript-[timestamp].md` として保存します。これには次を含めます。
- 収集したコンテキスト（何か、誰向けか、成功条件）
- 生のプレモーテム失敗理由
- すべての深掘りワーカーの分析
- 完全な統合

## 出力形式

すべてのプレモーテムセッションは2つのファイルを生成します。

```
premortem-report-[timestamp].html    # スキャン用のビジュアルレポート
premortem-transcript-[timestamp].md  # 参照用の完全なトランスクリプト
```

ユーザーには HTML レポートを最初に見せます。トランスクリプトは、各失敗シナリオの背後にある推論をより深く確認したい場合のためにあります。

また、チャットでは簡潔な要約を提供します。最も起こりやすい失敗、隠れた仮定、計画に対する最も重要な修正を、最大3文で述べます。詳細はレポートにあります。

## 例: プロダクトローンチをプレモーテムする

**ユーザー:** "premortem this: I'm about to launch a $297 live workshop on how to use an AI coworker tool for marketing teams. 50 seats. Targeting marketing managers at companies with 10-50 employees."

**生のプレモーテムが6つの失敗理由を特定する:**
1. この規模の会社のマーケティングマネージャーは、専門能力開発に $297 を使うために承認が必要であり、その摩擦をあなたは織り込んでいない
2. 「マーケティング向け AI coworker tool」はツール特化の売り文句だが、市場では多くのマネージャーがまだ AI が自分たちに関係あるのかを見極めている段階である
3. 実際に購入するオーディエンスはチームマネージャーではなくソロプレナーかもしれず、コンテンツと参加者のミスマッチが起きる
4. マーケティングチーム向けワークショップを作るには、現実的なマーケティングデータと複数席のセットアップを持つデモ環境が必要であり、見積もった2週間ではなく5週間の準備がかかる
5. 参加者の60%がソロプレナーだった場合、レビューやケーススタディは今後のコホートで必要なマーケティングマネージャー層に響かない
6. $297 で50席の場合、最大売上は $14,850 であり、他の収益機会と比べて準備時間に見合わない可能性がある

**6つの深掘りワーカーが各理由を独立に深掘りし、失敗の物語、根底にある仮定、早期警戒サインを生成する。**

**統合:** 最も起こりやすい失敗はオーディエンスのミスマッチです。あなたは $297 を使うために承認が必要な人たちをターゲットにしており、その摩擦を織り込んでいません。最も危険な失敗は、チームマネージャーではなくソロプレナーを集めてしまうことで、今後のコホートで本当に必要なターゲット購買者にケーススタディや推薦文が響かず、問題が時間とともに複利的に悪化することです。隠れた仮定は、「10〜50人規模の会社のマーケティングマネージャー」が到達可能なオーディエンスだと仮定していることです。しかし、彼らはそのように自己認識しておらず、同じ場所に集まっているわけでもありません。修正版の計画: まず20人向けに $47 のパイロットセッションを実施します。それを使って、実際の購買者がチームマネージャーなのかソロプレナーなのかを特定し、実際に来た人に合わせて本番ワークショップを作ります。

## 重要な注意

- **並列実行できる環境では、必ずすべての失敗深掘りを並列で実行する。** 逐次実行は時間を無駄にし、先の分析が後の分析に影響してしまいます。利用中の環境がサブエージェントを提供している場合はサブエージェントを使い、提供していない場合は独立した分析ブロックとして扱います。
- **必ずプレモーテムフレームを明示的に設定する。** 「これはすでに失敗した」という前提が、この手法を機能させる心理的メカニズムです。これがないと、分析は正直な失敗特定ではなく、丁寧なリスク評価に戻ってしまいます。
- **包括的に、ただし水増ししない。** 本物の失敗理由をすべて見つけます。7つあるのに3つで止めてはいけません。ただし、本当は3つしかないのに7つに増やしてはいけません。その数は、この具体的な計画に対して実際に存在するものにします。
- **統合がプロダクトである。** ほとんどのユーザーは統合を読み、個別の失敗カードはざっと確認します。統合は具体的で実行可能にします。
- **砂糖衣をかけない。** プレモーテムの要点は、現実が伝える前に、ユーザーが聞きたくないことを伝えることです。計画に深刻な問題があるなら、直接そう言ってください。
- **修正版の計画は具体的でなければならない。** 「価格をテストすることを検討する」と言ってはいけません。「本番の $297 ワークショップにコミットする前に、20人に対して $47 のパイロットを実施する」と言ってください。すべての修正は、ユーザーが今週実行できるものである必要があります。
- **最低コンテキスト基準を尊重する。** 不十分なコンテキストでプレモーテムを実行すると、一般的な失敗理由が出てユーザーの時間を無駄にします。悪いプレモーテムを出すより、もう1つ質問する方がよいです。
- **これは複数視点レビューではない。** 複数視点レビューは現在の意思決定に複数の視点を与えます。プレモーテムは、意思決定がすでに失敗した未来にAIエージェントを送り、そこから逆算して理由を説明させます。心理的メカニズムも出力も異なります。ユーザーが失敗分析ではなく複数視点を求めているようなら、代わりに複数視点レビューを提案してください。
