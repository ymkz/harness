# Autonomous

このスキルは、ユーザーが詳細な手順を指定せず、目的達成までの進め方をエージェントに委ねているときに使う。ゴールは「それっぽく進める」ことではなく、現在のリポジトリと制約を読んだうえで、実装、検証、報告まで完了させること。

## 基本姿勢

- まず現状を読む。対象ファイル、関連テスト、既存パターン、作業ツリーの未コミット変更を確認する。
- ユーザーのゴールを具体化する。曖昧さが残っても、合理的で低リスクな仮定を置けるなら止まらず進める。
- 既存の設計、命名、テスト方針に寄せる。不要な新規抽象化や大きなリファクタは避ける。
- 変更は依頼の範囲に閉じる。関係ない未コミット変更は触らない。
- 実装だけで終わらず、可能な限り検証まで行う。検証できなかった場合は理由を明確にする。

## 進め方

1. **状況把握**
   - `git status --short` で作業ツリーを確認する。
   - `rg`、`rg --files`、既存テスト、ドキュメントから関連箇所を探す。
   - 既存の実装スタイル、エラー処理、テストの置き方、生成物の扱いを確認する。

2. **作業計画**
   - 変更範囲、検証方法、リスクを短く整理する。
   - 大きめの作業ではチェックリスト化し、進捗に応じて更新する。
   - すぐに実装できる明確なタスクでは、長い計画で止まらず作業へ進む。

3. **実装**
   - 最小の安全な差分で目的を満たす。
   - 既存パターンに合うヘルパーや抽象化があるならそれを使う。
   - 新しい抽象化は、重複や複雑さを実際に減らす場合だけ追加する。
   - ユーザーが作った可能性のある変更を巻き戻さない。

4. **検証**
   - 変更に最も近いテスト、型チェック、lint、ビルドを優先して実行する。
   - 失敗した場合は、失敗が自分の変更由来か既存状態由来かを切り分ける。
   - 検証コマンドが重い、権限が必要、依存が不足している場合は、その制約を明記する。

5. **仕上げ**
   - 差分を読み返し、不要な変更、デバッグ出力、過剰なコメントを取り除く。
   - 変更点、検証結果、残リスクを簡潔に報告する。
   - コミットはユーザーが求めた場合だけ行う。

## サブエージェントの扱い

サブエージェントは、ユーザーが明示的に並列作業、委任、サブエージェント利用を求めており、かつ実行環境のルール上も許可されている場合だけ使う。

使う場合は、次を守る。

- 最初に自分で全体方針と直近のクリティカルパスを決める。
- サブエージェントには、互いに重ならない具体的な調査・実装・検証タスクを渡す。
- コード変更を任せる場合は、担当ファイルや責務を明示する。
- サブエージェントを検索エンジン代わりに使わない。曖昧な丸投げではなく、成果物が明確な仕事を渡す。
- 戻ってきた結果は鵜呑みにせず、最終的な統合と検証は自分で責任を持つ。

サブエージェントが使えない環境では、同じ考え方を単一エージェント内の独立した作業単位として扱う。調査、実装、検証、レビューを混ぜず、順に切り分けて進める。

## 完了基準

次を満たしたら完了としてよい。

- ユーザーの依頼に対応する差分が実装されている。
- 変更範囲に見合う検証を実行済み、または実行できない理由を説明できる。
- 関係ないファイルやユーザーの未コミット変更を巻き込んでいない。
- 最終報告に、変更内容、検証結果、残っている注意点が含まれている。
